From SQL pipelines to natural-language exploration

01 Where we are

Where we are today

Vatican Explorer, DSSV FY26 Data Conclave project.

  • Scrapes speeches, homilies, and audiences from vatican.va
  • Stores them in a local SQLite database
  • Enables analyses like “who cites 1 John 4 and how often?”

Working branch: agent. Goal of this talk: propose giving the corpus a natural-language front door, and decide together if it’s worth building.

Our SQL backbone

CREATE TABLE popes (
  _pope_id INTEGER PRIMARY KEY,
  pope_name TEXT, pope_number TEXT,
  secular_name TEXT, place_of_birth TEXT,
  pontificate_begin TEXT, pontificate_end TEXT,
  UNIQUE(pope_name, pope_number)
);

CREATE TABLE texts (
  _texts_id INTEGER PRIMARY KEY,
  pope_id INTEGER, section TEXT, year TEXT,
  date TEXT, location TEXT, title TEXT,
  language TEXT, url TEXT, text_content TEXT,
  FOREIGN KEY (pope_id) REFERENCES popes(_pope_id)
);

The friction today

To ask: “How did Benedict XVI talk about mercy in 2010?” a user must:

conn, cur = connect_to_database()
register_regexp_function(conn)
cur.execute("""
  SELECT t.title, t.date, t.text_content
  FROM texts t JOIN popes p
  ON t.pope_id = p._pope_id
  WHERE p.pope_name = ?
    AND t.year = ?
    AND t.text_content REGEXP ?
""", ("Benedict XVI", "2010", r"mercy|misericord"))

Warning

Requires knowing

  • The schema
  • Python + SQL
  • Regex
  • Which notebook to open

Not accessible to historians or theologians.

02 Why an agent?

The proposal

Same backend. New front door.

  • User asks a question in plain English.
  • An agent would pick the right queries, run them against our existing DB, and return a grounded answer with citations.
  • No new scraping, no new schema; we reuse what we already built.
  • Today is about deciding if this is worth doing, not about a finished product.

What is an agent?

An agent is an LLM in a loop that can call tools.

Chatbot

  • One shot
  • Text in, text out
  • Cannot touch your data
  • Can make things up

Agent

  • Runs in a loop
  • Decides which tool to call
  • Reads real results
  • Iterates until done

The LLM is the planner. Your functions are the hands.

The agent loop

Every hop is transparent: we can log it, test it, and debug it.

Tools are just functions

We already wrote most of them.

# database_helpers.py (existing)
def fetch_rows_by_regexp(cursor, query, pattern): ...

# search_biblical_citation.py (existing)
def search_biblical_citations_db(pattern, query): ...

Turn them into agent tools with a Pydantic schema:

from pydantic import BaseModel, Field

class SearchTextsInput(BaseModel):
    pope:    str | None = Field(None, description="e.g. 'Francis'")
    section: str | None = Field(None, description="homilies | audiences | ...")
    year:    str | None = None
    regex:   str | None = Field(None, description="Python regex over text_content")

We have already written 90% of the agent. The LLM just picks which one to run.

03 The tech

Tech stack: the choices

Layer Options
LLM provider Anthropic Claude · OpenAI · Ollama (local)
Agent framework Raw SDK loop · PydanticAI · LangChain · LlamaIndex
Validation Pydantic
Vector store (optional) sqlite-vec · Chroma
CLI python -m dc26_vatican_explorer.agent

Evaluation axes: stability, lock-in, typed I/O, deps, offline story.

Tech stack: our pick

Tip

Anthropic Claude + PydanticAI + Pydantic

Why:

  • First-class tool use that is stable and well-documented
  • Typed tool I/O matches our existing function signatures naturally
  • Minimal new dependencies, no heavy framework lock-in
  • Fits a clean pyproject.toml optional group:
[project.optional-dependencies]
agent = ["anthropic", "pydantic-ai", "pydantic>=2"]

Fallbacks if constraints change:

  • OpenAI + raw SDK. If we need OpenAI-specific features.
  • Ollama + local model. If we go fully offline or zero-cost.

04 Discussion

If we say yes: possible roadmap

Milestones (for discussion)

  • M1. Tool wrappers + unit tests (mirror tests/test_database_helpers.py)
  • M2. LLM loop + CLI entry point
  • M3. Evaluation set: 20 gold Q/A pairs, regression-tested
  • M4. Optional semantic search via embeddings (sqlite-vec)

Risks to watch

  • Hallucinated SQL or citations → mitigate with typed tools + grounded answers
  • Cost & latency → cache tool results, cap loop iterations
  • Evaluation → build the gold set before we tune prompts

Open questions for the group

  1. Is this the right next feature? Or should effort go into pope_comparison/ instead?
  2. Which LLM provider? Cloud (Claude/OpenAI) vs. local (Ollama): cost, privacy, reproducibility tradeoffs.
  3. Framework: PydanticAI, raw SDK, or LangChain? How much abstraction do we want?
  4. Who is the user? Us, other researchers, or a public-facing demo? Changes UX and security.
  5. How do we evaluate it? What does “good enough” look like for a humanities corpus?
